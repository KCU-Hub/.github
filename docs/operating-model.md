# KCU-Hub 운영 모델

## 목적

KCU-Hub는 학부, 동아리, 스터디, 프로젝트 활동에서 생기는 지식을 다음 사람에게 넘길 수 있게 정리하는 GitHub org입니다. GitHub는 커뮤니티 채팅방이 아니라 산출물 저장소, 협업 기록, 공개 문서 허브로 사용합니다.

## 핵심 판단

| 항목 | 판단 | 이유 |
|---|---|---|
| 메인 커뮤니티 | 기존 풀스택 커뮤니티 앱을 메인으로 쓰지 않습니다. | 앱 운영은 인증, moderation, 데이터 관리 부담이 큽니다. 지식 전수 목표에는 GitHub 문서와 repo 흐름이 더 가볍습니다. |
| 실시간 소통 | 외부 채널을 씁니다. | 빠른 질문, 공지, 참여 독려는 채팅형 채널이 더 적합합니다. |
| 장기 기록 | GitHub에 남깁니다. | README, Issue, PR, Discussion, Pages는 검색과 이력 추적에 강합니다. |
| repo 확장 | 유지보수자가 생길 때만 늘립니다. | 빈 repo가 많아지면 신뢰가 떨어지고 정리 비용이 커집니다. |
| 공개 정보 | 개인 정보와 민감 정보를 제외합니다. | 공개 repo는 검색되고 복제될 수 있으므로 안전한 공개 범위만 남깁니다. |

## 운영 레이어

| 레이어 | 맡는 일 | 대표 repo 또는 기능 |
|---|---|---|
| Org profile | KCU-Hub의 정체성, 참여 경로, 운영 원칙 | `.github` |
| Public docs | 읽기 좋은 공개 문서와 안내 사이트 | `hub-docs` 또는 `KCU-Hub.github.io` |
| Curriculum | 학기별, 주제별 학습 로드맵 | `curriculum-roadmap` |
| Notes | 과목별 기록과 참고 링크 | `course-notes` |
| Workshops | 실습 자료, 템플릿, 발표 자료 | `workshops` |
| Projects | 만들 거리, 난이도, 담당자 없는 제안 | `project-ideas` |
| Curation | 좋은 자료와 repo 모음 | `awesome-cuk-sw` |

## 최소 운영 흐름

1. 외부 채널에서 빠르게 논의합니다.
2. 오래 남길 내용만 GitHub Issue나 Discussion으로 옮깁니다.
3. 실제 문서 변경은 PR로 남깁니다.
4. 운영자는 주 1회 Issue와 PR을 정리합니다.
5. 학기 말에는 남은 Issue, 오래된 문서, 다음 학기 로드맵을 정리합니다.

## 새 repo 생성 기준

새 repo는 다음 질문에 모두 답할 수 있을 때 만듭니다.

| 질문 | 기준 |
|---|---|
| 무엇을 보관하나요? | 첫 3개 산출물이 명확해야 합니다. |
| 누가 정리하나요? | 최소 1명의 유지보수자가 있어야 합니다. |
| 얼마나 자주 바뀌나요? | 월 1회 이상 업데이트할 이유가 있어야 합니다. |
| 기존 repo와 겹치나요? | 문서 한 폴더로 충분하면 새 repo를 만들지 않습니다. |
| 공개해도 되나요? | 개인 정보, 계정 정보, 민감 정보가 없어야 합니다. |

## 남겨둘 것

- 과목별 공개 가능한 정리 노트
- 스터디 커리큘럼과 회차별 결과
- 프로젝트 README, 설치 방법, 회고
- 발표 자료, 실습 자료, 템플릿
- 운영 규칙과 변경 이력

## 하지 말아야 할 것

- 공개 repo에 개인 정보, 이메일, 계정 정보, 내부 식별자를 남기지 않습니다.
- GitHub Issue를 실시간 채팅처럼 쓰지 않습니다.
- 합의되지 않은 전략이나 KPI를 문서에 확정 표현으로 넣지 않습니다.
- repo를 많이 만드는 것 자체를 성장 지표로 삼지 않습니다.
- 기존 커뮤니티 앱을 KCU-Hub 전체의 운영 중심으로 격상하지 않습니다.
