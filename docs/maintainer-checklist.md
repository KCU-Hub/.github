# 운영자 체크리스트

## 주간

| 확인 항목 | 기준 |
|---|---|
| 새 Issue triage | 목적, 공개 가능 여부, 완료 기준이 있는지 확인합니다. |
| 새 PR review | 문서가 실제 파일 구조와 맞는지 확인합니다. |
| 깨진 링크 | README와 주요 docs 링크를 확인합니다. |
| 외부 채널 반복 질문 | GitHub에 남길 FAQ인지 판단합니다. |

## 월간

| 확인 항목 | 기준 |
|---|---|
| repo 목록 | 빈 repo, 중복 repo, 방치 repo를 확인합니다. |
| 문서 최신성 | 오래된 학기 정보나 미완성 문구를 정리합니다. |
| Projects board | 완료된 카드와 막힌 카드를 정리합니다. |
| 공개 범위 | 개인정보, 계정 정보, 내부 식별자가 없는지 확인합니다. |

## 학기 말

| 확인 항목 | 기준 |
|---|---|
| 로드맵 정리 | 다음 학기 사람이 볼 수 있게 완료/보류를 나눕니다. |
| 과목 노트 정리 | 공개 가능한 요약과 참고 링크만 남깁니다. |
| 프로젝트 회고 | 완료 repo, 중단 repo, 다음 과제를 연결합니다. |
| 새 유지보수자 인수인계 | repo별 담당자와 정리 주기를 문서화합니다. |

## PR review 기준

- README의 설명이 실제 파일과 맞아야 합니다.
- 새 문서는 다음 사람이 바로 따라갈 수 있어야 합니다.
- 출처나 외부 링크가 있으면 링크가 열려야 합니다.
- 개인 정보, 계정 정보, 내부 식별자가 없어야 합니다.
- 합의되지 않은 전략, KPI, 공식 입장을 확정 표현으로 쓰지 않습니다.

## repo 생성 review 기준

| 질문 | 통과 기준 |
|---|---|
| 첫 3개 산출물이 있나요? | 파일명 또는 Issue로 구체화되어 있습니다. |
| 유지보수자가 있나요? | 최소 1명이 정리 주기를 맡습니다. |
| 기존 repo와 겹치지 않나요? | 폴더나 문서로 충분하지 않습니다. |
| 공개 repo로 안전한가요? | 민감 정보 제거 기준이 있습니다. |
| 3개월 뒤에도 읽힐 가치가 있나요? | 일회성 공지보다 누적 지식에 가깝습니다. |
