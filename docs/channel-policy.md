# 채널 역할 분담

## 원칙

GitHub는 오래 남길 기록을 맡고, 외부 채널은 빠른 소통을 맡습니다. 같은 내용을 두 곳에 모두 길게 쓰지 말고, 외부 채널에서 합의된 결과만 GitHub 문서나 Issue로 정리합니다.

## 역할 표

| 채널 | 적합한 일 | 남기지 말아야 할 것 | GitHub로 옮기는 기준 |
|---|---|---|---|
| GitHub README/docs | 확정된 안내, 로드맵, 과목 노트, 프로젝트 문서 | 실시간 잡담, 개인정보, 미확정 공지 | 다음 사람도 읽어야 하는 내용입니다. |
| GitHub Issue | 고쳐야 할 문서, 새 repo 제안, 작업 단위 | 단순 질문, 출석 확인, 민감한 상담 | 담당자와 완료 기준이 있습니다. |
| GitHub Discussion | 오래 남길 질문, 스터디 회고, 자료 추천 | 즉답성 대화, 개인 사정 | 나중에 검색할 가치가 있습니다. |
| GitHub Projects | 운영자 triage, 학기별 마일스톤, PR 상태 | 개인별 평가, 비공개 관리 정보 | 공개 작업 흐름으로 볼 수 있습니다. |
| GitHub Pages | 공개 안내 사이트, repo index, 참여 가이드 | 빠른 공지, 임시 자료 | 방문자가 처음 볼 자료입니다. |
| 카카오톡/밴드/네이버 카페 | 공지, 빠른 질문, 참여 독려, 실시간 조율 | 오래 남길 최종 문서 | 반복 질문이 되거나 다음 기수에게 필요합니다. |

## GitHub 기능 사용 제안

| 기능 | 권장 사용 | 비고 |
|---|---|---|
| Discussions | `hub-docs` 또는 `course-notes`에서 중앙 Q&A와 자료 추천을 받습니다. | repo가 생긴 뒤 카테고리를 `Q&A`, `스터디`, `자료 추천`, `운영 제안` 정도로 제한합니다. |
| Issues | 모든 repo에서 작업 가능한 단위만 받습니다. | 템플릿으로 목적, 공개 가능 여부, 완료 기준을 확인합니다. |
| Projects | org 운영자가 하나의 public board로 관리합니다. | 상태는 `Backlog`, `Ready`, `In progress`, `Review`, `Done` 정도로 충분합니다. |
| Pages | `hub-docs` 한 곳에서만 운영합니다. | 여러 repo Pages는 보류합니다. |
| Wiki | 기본적으로 쓰지 않습니다. | PR 리뷰와 이력 관리가 약하므로 Markdown 문서를 우선합니다. |

## 외부 채널에서 GitHub로 옮기는 예시

| 외부 채널 내용 | GitHub 기록 방식 |
|---|---|
| 같은 질문이 세 번 이상 반복됩니다. | FAQ 문서 또는 Discussion으로 정리합니다. |
| 스터디 커리큘럼이 정해졌습니다. | `curriculum-roadmap` PR로 남깁니다. |
| 워크숍 자료가 배포되었습니다. | `workshops`에 README와 자료 링크를 남깁니다. |
| 프로젝트 아이디어가 나왔습니다. | `project-ideas` Issue로 옮겨 난이도와 산출물을 적습니다. |
| 오류 제보가 들어왔습니다. | 해당 repo Issue로 옮겨 재현 방법을 적습니다. |
